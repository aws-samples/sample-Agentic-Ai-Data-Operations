# Orchestrator Examples - Demo Only

This folder contains **example DAGs that coordinate multiple workloads**. These are NOT used in generic data onboarding.

---

## Generic Pattern (Production)

In production, **each workload has its own independent DAG**:

```
workloads/customer_master/dags/customer_master_dag.py
workloads/order_transactions/dags/order_transactions_dag.py
workloads/sales_transactions/dags/sales_transactions_dag.py
```

These are generated by the **Data Onboarding Agent** when onboarding a new customer dataset. Each DAG:
- Runs Bronze → Silver → Gold pipeline for ONE workload
- Has its own schedule (daily, hourly, etc.)
- Is independent and self-contained

---

## What's in This Folder (Demo Examples)

### `end_to_end_pipeline_dag.py`
**Purpose**: Demo showing how to coordinate 2 workloads with dependencies

**Hardcoded for**:
- `customer_master` (runs first)
- `order_transactions` (waits for customer_master to complete FK validation)

**Use case**: When you have multiple workloads with dependencies (e.g., orders need customers for FK lookup)

**Production equivalent**: Set dependencies in `config/schedule.yaml`:
```yaml
dependencies:
  - workload: customer_master
    zone: staging  # Wait for customer staging before processing orders
```

### `mcp_orchestrator_dag.py`
**Purpose**: Dynamic orchestrator that auto-discovers workloads

**Features**:
- Reads all `workloads/*/config/` to discover datasets
- Creates pipeline stages dynamically
- Routes operations through MCP servers
- Generates agent-aware logs

**Use case**: Single DAG that manages all workloads (alternative to individual DAGs)

**Trade-offs**:
- ✅ One DAG to maintain
- ❌ All workloads share the same schedule
- ❌ Harder to debug individual workload failures

### `MCP_ORCHESTRATOR_GUIDE.md`
Documentation for deploying the MCP orchestrator to MWAA.

## When to Use These

### Use Individual DAGs (Generic Pattern) When:
- ✅ Each workload has different schedule (some daily, some hourly)
- ✅ Workloads are independent (no FK dependencies)
- ✅ Different teams own different workloads
- ✅ Need to pause/modify one workload without affecting others

**This is the recommended pattern for production.**

### Use Orchestrator DAG When:
- Multiple tightly-coupled workloads (FK relationships)
- All workloads run on same schedule
- Want single point of control
- Demo/testing scenarios

## Production Data Onboarding

When the Data Onboarding Agent onboards a new customer dataset:

1. **Agent asks**: "What's your data source?" (PostgreSQL, S3, API, etc.)
2. **Agent generates**: `workloads/{dataset_name}/dags/{dataset_name}_dag.py`
3. **DAG deployed**: User uploads to MWAA S3 bucket
4. **Runs independently**: No orchestrator needed

**No manual DAG coordination required.**

---

## How Dependencies Work (Generic Pattern)

If a workload depends on another (e.g., orders need customers):

**Option 1**: Use Airflow ExternalTaskSensor in the dependent DAG
```python
wait_for_customers = ExternalTaskSensor(
    task_id="wait_for_customers",
    external_dag_id="customer_master_dag",
    external_task_id="publish_customers",
)
```

**Option 2**: Specify dependencies in `config/schedule.yaml`
```yaml
schedule:
  frequency: daily
  start_time: "06:00"
  dependencies:
    - workload: customer_master
      zone: staging
```

The Data Onboarding Agent generates the sensor code automatically.
